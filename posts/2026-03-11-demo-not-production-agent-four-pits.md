今天很多团队都在晒 Agent 跑通截图：能调工具、能自动执行、还能串多步流程。

我先把结论放在最前面：**会跑 Demo 只是入场券，能上生产才是分水岭。**

所谓“80% 团队会死在同一个坑”，不是统计口径，而是我这段时间看到最高频、最致命的工程现实：

- 默认配置直接上线上
- 权限边界没收
- 失败路径没设计
- 监控告警没打通

结果就是一句话：**顺风很惊艳，逆风全崩盘。**

## 为什么“把 Demo 当生产”这么容易发生

因为 Demo 的激励是“快跑通”，生产的激励是“稳交付”。

Demo 阶段你只需要回答一个问题：
- 它能不能工作一次？

生产阶段你必须回答五个问题：
- 它能不能稳定工作一万次？
- 出错时能不能止损？
- 故障时谁接管？
- 操作能不能审计追责？
- 三个月后还能不能维护？

很多团队的问题不是“不会做 Agent”，而是把第一套问题错当成了第二套问题。

## 坑 1：把默认权限当生产权限（最蠢也最常见）

最常见的爆炸路径是：为了先跑通，把工具权限开到最大；跑通后没人再关回来。

短期你会觉得效率很高，长期你会得到三种风险：

1. **误操作放大**：模型一次误判就能触发高风险动作
2. **权限污染**：低风险任务拿到了高风险能力
3. **责任不清**：事后很难说清楚到底谁触发了什么

可执行改法很简单：

- 按任务分层权限，不按“角色感觉”给权限
- 高风险动作默认人工确认（删除、转账、外发）
- 所有工具调用必须留参数与结果日志

别把“先开着再说”当临时方案，它通常会变成永久事故。

## 坑 2：没有可观测性，系统在“盲飞”

很多 Agent 项目有成功截图，没有失败画像。

你问它哪里慢、哪里错、哪里抖，团队只能靠体感回答。这不叫系统，这叫玄学。

生产最小观测集至少要有：

- **链路级耗时**：每一步耗时、总耗时、超时分布
- **结果级质量**：成功率、失败率、重试率、人工接管率
- **成本级数据**：token 消耗、工具调用成本、单位任务成本
- **风险级事件**：越权尝试、敏感操作、回滚次数

没有这些，你连“优化”都谈不上，因为你根本不知道在优化什么。

## 坑 3：失败路径没设计，异常时全靠人扛

几乎所有 Demo 默认假设“外部依赖都可用”。

但真实世界会发生：
- 模型抖动
- API 限流
- 上游数据脏值
- 工具返回格式突变

如果没有失败路径，系统会在压力下表现为：
- 无限重试
- 雪崩排队
- 错误扩散

建议的最小失败治理：

1. 给每条关键链路设置超时与重试上限
2. 对重试做指数退避，不要硬冲接口
3. 关键节点支持断路与降级
4. 预留人工接管通道，一键切回人控

记住：**生产系统不是“不会失败”，而是“失败后还能活”。**

## 坑 4：缺少变更治理，修一个点炸一片面

Agent 工作流天然是“高耦合系统”：模型提示词、工具定义、上下文结构、权限策略彼此联动。

你改一行提示词，可能影响十个下游动作。

如果没有变更治理，常见后果是：
- 今天修 A，明天坏 B
- 回归范围不清楚
- 故障复盘只有口头结论

最低限度你得有：

- 版本化的 Prompt / Tool Schema
- 变更前后对比测试（至少覆盖关键路径）
- 灰度发布与快速回滚机制
- 复盘模板：触发条件、影响面、止损动作、后续修复

不做治理的自动化，迟早会变成“自动制造事故”。

## 一个 7 天可落地的生产化修复方案

### Day 1-2：先收边界
- 权限最小化
- 高风险动作加人工确认
- 明确“禁止自动执行”清单

### Day 3-4：补观测
- 打通链路日志
- 建立基础告警（失败率、超时、重试风暴）
- 记录人工接管原因

### Day 5：做故障演练
- 模拟模型不可用、API 限流、上下文污染
- 记录恢复时长与业务影响

### Day 6：补回滚与灰度
- 建立配置版本
- 关键流程支持一键回退

### Day 7：复盘并固化
- 输出故障手册
- 更新发布检查清单
- 明确值班与责任边界

这 7 天做完，你还不一定“很先进”，但至少不会“很脆弱”。

## 我的判断

Agent 这波红利是真的，但市场会很快从“能不能做出来”切换到“能不能稳定交付”。

真正能穿越周期的团队，不是提示词写得最花的团队，而是把这四件事做扎实的团队：

- 权限边界
- 可观测性
- 失败治理
- 变更治理

一句话收尾：**Demo 证明可能性，生产决定生死线。**
